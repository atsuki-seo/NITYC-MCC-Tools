# シラバス Markdown スキーマ

`class-syllabus` が出力し、`class-syllabus-parse` が読み取るシラバス Markdown の**入出力契約**。`class-test` / `class-report` などの後続スキルも、スケジュール情報や名前の正規化ルールを本契約に従って解釈する。

このファイルは「出力仕様（生成時のルール）」と「入力仕様（抽出時のルール）」を一つに束ねた**単一の正本**である。各スキルのリファレンスに同じルールを二重記述せず、必ず本ファイルを参照すること。

## 参照元スキル

- `class-syllabus`: Step 5・Step 5.5・Step 8 の Markdown 生成で本スキーマに従って出力する
- `class-syllabus-parse`: Step 2〜4 の抽出で本スキーマに従って読み取る
- `class-test`: 試験名の正規化形と試験スケジュールの解釈で本スキーマに従う
- `class-report`: レポート名の正規化形とレポートスケジュールの解釈で本スキーマに従う

---

## 1. 全体構造

シラバス Markdown は以下のセクション構成で出力する。`class-syllabus-parse` は各セクションを見出しレベルで識別して抽出する。

```markdown
# シラバス: [教科名]

## 基本情報
（表形式）

## 到達目標
（本文）

## ルーブリック
（表形式）

## 概要
（本文）

## 授業の進め方
（本文 + 構造化サブセクション）

### レポート・課題スケジュール
（表形式。レポート・成果物が評価に含まれる場合のみ）

### 試験スケジュール
（表形式。小テストまたは定期試験が評価に含まれる場合のみ）

## 注意点
（本文）

## 授業計画
（表形式）

## 評価割合
（表形式 + 計算方法・計算例）

## MCC対応
（表形式。MCC対応付けを行った場合のみ）

## 授業の属性
（表形式）
```

## 2. 基本情報

### 2.1 出力仕様

`## 基本情報` セクションで以下を表形式で出力する。

| 項目 | 必須 | 備考 |
|------|------|------|
| 教科名 | ○ | |
| 開講年度 | ○ | `[YYYY]年度` 形式 |
| 担当教員 | ○ | |
| 対象学年 | ○ | |
| 開講期 | ○ | |
| 単位 | ○ | |
| 教科書/教材 | ○ | URL または書名 |
| 科目番号 | △ | シラバスに存在する場合のみ |
| 科目区分 | △ | |
| 授業形態 | △ | |
| 開設学科 | △ | |
| 週時間数 | △ | |

### 2.2 抽出仕様

`class-syllabus-parse` は `## 基本情報` 見出し以下の表、または冒頭セクションの `項目: 値` 形式から以下を抽出する。

| 項目 | 抽出方法 |
|------|---------|
| 科目名 | `教科名:` / `科目名:` / 文書タイトル（h1）から取得 |
| 担当教員 | `担当教員:` 行 |
| 対象学年 | `対象学年:` 行 |
| 開講期 | `開講期:` 行 |
| 単位 | `単位:` 行 |
| 評価割合 | `## 評価割合` 表から取得 |

見つからない項目は「（記載なし）」として扱う（エラーにはしない）。

## 3. スケジュール（構造化サブセクション）

レポート・試験のスケジュールは、`## 授業の進め方` セクションの末尾（`## 注意点` の手前）に**構造化テーブル**として出力する。これらのテーブルは後続スキルが機械的に抽出する際の入口となる。

### 3.1 レポート・課題スケジュール

**出力条件**: レポート・成果物が評価に含まれる場合に挿入する。

**列仕様（固定3列）**:

| 列名 | 内容 | 形式 |
|------|------|------|
| `レポート名` | 正規化済みのレポート名 | § 5.1 参照 |
| `提出週` | 提出週 | `第[N]週` |
| `テーマ` | 学生が何に取り組むかが分かる具体的な名前 | 自由テキスト |

**出力例**:

```markdown
### レポート・課題スケジュール
| レポート名 | 提出週 | テーマ |
|-----------|--------|-------|
| レポート1 | 第8週 | 字句解析の設計とプログラミング実践 |
| レポート2 | 第11週 | 構文解析の設計とプログラミング実践 |
| 最終レポート | 第14週 | 独自プログラミング言語の設計と実装 |
```

**出力時の注意**:
- 箇条書き本文とテーブルの内容は整合させる（同じレポート数・同じ提出週・同じテーマ）
- `レポート名` は § 5.1 の正規化ルールに従って出力時に正規化する

### 3.2 試験スケジュール

**出力条件**: 小テストまたは定期試験が評価に含まれる場合に挿入する。

**列仕様（固定3列）**:

| 列名 | 内容 | 形式 |
|------|------|------|
| `試験名` | 正規化済みの試験名 | § 5.2 参照 |
| `実施週` | 実施週 | `第[N]週` |
| `範囲` | 出題範囲 | `第[X]〜[Y]週` または `全範囲（第[X]〜[Y]週）` |

**出力例**:

```markdown
### 試験スケジュール
| 試験名 | 実施週 | 範囲 |
|--------|--------|------|
| 小テスト1 | 第5週 | 第1〜4週 |
| 小テスト2 | 第8週 | 第5〜7週 |
| 期末試験 | 第15週 | 全範囲（第1〜14週） |
```

**出力時の注意**:
- 箇条書き本文とテーブルの内容は整合させる
- `試験名` は § 5.2 の正規化ルールに従って出力時に正規化する
- 範囲が確定していない時点では `（要確認）` と記載してよい（`class-syllabus` Step 5.5 で確定させる）

## 4. スケジュールの抽出ロジック

`class-syllabus-parse` はスケジュールを以下の優先順位で抽出する。

### 4.1 優先パス: 構造化テーブルからの読み取り

§ 3.1 / § 3.2 のテーブルを最優先で読み取る。

**前提**:
- 列構成は固定（レポート: 3列 / 試験: 3列）
- 名前は既に § 5 の正規化ルールに従って正規化済み
- 週表記は `第[N]週` 形式
- 試験の範囲は `第[X]〜[Y]週` または `全範囲（第[X]〜[Y]週）` 形式
- 合計行や注釈行がテーブル内にある場合は除外する

### 4.2 フォールバック: フリーテキストからのパターンマッチ

構造化テーブルが無い既存シラバス（本スキーマ策定前に生成されたもの）に対しては、`## 授業の進め方` 本文のパターンマッチで抽出する。

#### レポートのパターンマッチ

**積極パターン（採用する）**:
- `・レポート[丸数字または数字][:：][テーマ]（第[N]週提出）`
- `・[特殊名]レポート[:：][テーマ]（第[N]週提出）` — 特殊名: `中間`, `期末`, `最終`
- `・レポート[N]は第[N]週に提出`
- `レポート[丸数字または数字]（第[N]週）`

**否定パターン（採用しない）**:
- 「レポート返却」「レポート講評」「レポート解説」を含む行
- 「期限内に提出すること」「遅延提出は減点」など一般的な提出ルールの言及
- 「レポートにフィードバックを返す」などのメタ記述

#### 試験のパターンマッチ

**検出対象キーワード**: 「小テスト」「中間試験」「期末試験」「定期試験」「確認テスト」

**積極パターン（採用する）**:
- `・小テスト[丸数字または数字][:：]第[N]週（範囲: 第[X]〜[Y]週）`
- `・[中間|期末|定期]試験[:：]第[N]週（範囲: 第[X]〜[Y]週）`
- `小テスト[丸数字または数字]（第[N]週、範囲: 第[X]〜[Y]週）`
- `第[N]週に小テスト[丸数字または数字]（範囲: 第[X]〜[Y]週）`

**否定パターン（採用しない）**:
- 「試験の返却」「試験解説」「試験後のフィードバック」を含む行
- 「試験は実施しない」「試験は行わない」などの否定文
- 「試験対策として」「試験準備として」などのメタ記述

#### 範囲記載の解釈

| シラバス表記 | 解釈 |
|-------------|------|
| `第X〜Y週` / `第X週〜第Y週` / `第X-Y週` | そのまま範囲として使用 |
| `全範囲` / `全内容` / `これまでの内容` | 第1週〜当該試験の前週 |
| `前半` / `後半` など曖昧な表現 | 抽出失敗扱いにする |

抽出失敗時は、試験名と実施週だけ抽出し、範囲は「（要確認）」として出力する。後続スキルで教員に確認を求める。

### 4.3 補完: 授業計画からの補完

`## 授業計画` テーブルの `授業内容` 列に「レポート[N]作成/提出」「小テスト①（範囲: 第1〜4週）」などの記述がある場合、取りこぼしたスケジュール情報の補完として使う。ただし構造化テーブル・`## 授業の進め方` 本文で抽出した情報を優先する。

## 5. 名前の正規化ルール

レポート名・試験名は、**`class-syllabus` が出力時に以下の形式に正規化する**。`class-syllabus-parse` および後続スキルは、構造化テーブルの名前は既に正規化済みであることを前提に読み取る。

この正規化した名前を、後続スキル（`class-test` / `class-report` 等）がディレクトリ名・ファイル名・プレビュー表示で一貫して使用する。

### 5.1 レポート名

| ソース表記 | 正規化後 |
|-----------|---------|
| `レポート①`, `レポート1`, `第1レポート`, `第1回レポート` | `レポート1` |
| `レポート②`, `レポート2` | `レポート2` |
| `レポート③`, `レポート3` | `レポート3` |
| `課題A` | `レポート1`（英字→数字、出現順） |
| `課題B` | `レポート2` |
| `中間レポート` | `中間レポート` |
| `期末レポート` | `期末レポート` |
| `最終レポート` | `最終レポート` |

### 5.2 試験名

| ソース表記 | 正規化後 |
|-----------|---------|
| `小テスト①`, `小テスト1`, `第1回小テスト` | `小テスト1` |
| `小テスト②`, `小テスト2` | `小テスト2` |
| `小テスト③`, `小テスト3` | `小テスト3` |
| `中間試験`, `中間テスト` | `中間試験` |
| `期末試験`, `期末テスト` | `期末試験` |
| `定期試験` | `期末試験`（期末試験として扱う、中間と区別される場合のみそのまま） |
| `確認テスト[N]` | `小テスト[N]` |

### 5.3 共通ルール

1. 丸数字（①②③…）・漢数字（一二三…）は半角アラビア数字に変換する
2. 複数の同種テスト・レポートが存在する場合は出現順で連番にする
3. 単独のテスト・レポート（「中間試験」「期末レポート」など）はシラバスの記載をそのまま使用する
4. 「第N回」表記は末尾の数字のみ取り出し、「小テスト1」「レポート1」形式に統一する
5. ファイルシステムで使えない文字（`/`, `:`, `\`, 空白連続）はハイフンに置換する

## 6. 到達目標・ルーブリック

### 6.1 出力仕様

```markdown
## 到達目標
（散文形式または箇条書き。class-syllabus Step 2.5a のルールに従う）

## ルーブリック
| 評価観点 | 理想的 | 標準的 | 未到達 |
|----------|--------|--------|--------|
| ... | ... | ... | ... |
```

### 6.2 抽出仕様

- **到達目標**: `## 到達目標` 見出し以下の本文（段落または箇条書き）をそのまま取得する。形式は問わない
- **ルーブリック**: `## ルーブリック` 見出し以下の表を取得する。列数・列名が異なるテンプレートもあり得るため、表をそのまま読み取って保持する（列名の改変はしない）

---

## 7. 抽出結果のユーザー確認

`class-syllabus-parse` の Step 5 で、Step 2〜4 の抽出結果をまとめて提示し、`AskUserQuestion` で確認する:

- options: 「この内容で進む（後続スキルで使用する）」「抽出内容を修正したい」

「修正したい」が選ばれた場合、プレーンテキストで修正箇所を聞き、該当項目のみ差し替える。スケジュールの追加・削除もここで対応する。

---

## 8. 抽出が失敗した場合

以下のいずれかに該当する場合は、自動抽出を諦めて手動入力を求める:

- 基本情報（科目名・担当教員・対象学年）がすべて取得できない
- スケジュールの構造化テーブル・フォールバックの両方で記載を検出できず、かつ評価割合上はレポート・試験の配点がある

プレーンテキストで質問する:

```
シラバス情報を自動抽出できませんでした。以下の情報を教えてください:
- 科目名:
- レポート/試験の名前・実施週・範囲（該当するもの）:
- ...
```

複数項目がある場合は繰り返し入力を受け付ける。

## 9. parse のコンテキスト出力契約

`class-syllabus-parse` は実行後、会話コンテキストに以下の情報を残す。後続スキル（`class-test` / `class-report` / `class-slides` 等）はこの契約に依拠して動作する前提のため、parse はこれらの情報を必ずユーザーに表示し、コンテキストに保持されている状態でスキルを終了する。

### 9.1 コンテキストに残す情報

| カテゴリ | 含まれる項目 | 抽出元（本スキーマ内のセクション） |
|---------|-------------|----------------------------------|
| 基本情報 | 科目名・担当教員・対象学年・開講期・単位・評価割合 | § 2 |
| レポート・課題スケジュール | レポート名（正規化済み）・提出週・テーマ | § 3.1 / § 5.1 |
| 試験スケジュール | 試験名（正規化済み）・実施週・範囲 | § 3.2 / § 5.2 |
| 到達目標 | 到達目標本文 | § 6 |
| ルーブリック | ルーブリック表（評価観点・3段階記述） | § 6 |
| 授業計画 | 各週の授業内容・到達目標対応 | （シラバス本文の `## 授業計画` 表をそのまま保持） |
| シラバス生Markdown | シラバスファイルの全文 | （Read 結果） |

### 9.2 後続スキルでの利用

後続スキルは以下の方針で本契約を扱う:

- **再 Read しない**: 既にコンテキストに残っている情報を直接参照する
- **Step 0 で parse 実行を案内**: コンテキストに parse の出力が残っていない場合、後続スキルは `/class-syllabus-parse` の事前実行をユーザーに案内する
- **必要項目が欠ける場合の対応**: § 8 のフォールバック（手動入力）が parse 側で実行されているため、後続スキルは欠落しても処理を継続する。本当に必要な項目が欠けていた場合のみ後続スキル側で追加質問する

### 9.3 後続スキルへの呼び出し方の案内（parse 側の責任）

`class-syllabus-parse` の処理完了時には、抽出結果がコンテキストに残っている状態を確認し、後続スキル（`class-test` / `class-report` 等）の利用を案内してよい。明示的に何のスキルを呼ぶべきかの強制はしない（ユーザーの自由）。

### 9.4 契約変更時の影響範囲

§ 9.1 の表に項目を追加・削除・名称変更する場合、以下のスキルの該当箇所も同期する必要がある:

- `class-syllabus-parse`: Step 2〜4 の抽出・表示
- `class-test`: Step 0 の前提情報リスト
- `class-report`: Step 0 の前提情報リスト
- `class-slides`: Step 0 の前提情報リスト
