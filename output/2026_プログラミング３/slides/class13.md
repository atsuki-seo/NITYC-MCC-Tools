# 動作検証の観点

実装後に必ず通すべき確認:

| 観点 | 内容 |
|------|------|
| 正常系 | 想定通りの使い方で動くか |
| 異常系 | 空入力・不正値でエラーが出るか |
| 境界値 | 0 / 最大値 / 1文字 / 限界の長さ |
| 権限 | ログイン前/後の挙動の違い |
| 回帰 | 以前動いていた機能が壊れていないか |

---

# バグを見つけたら

1. **再現手順** を書く（環境・操作・期待挙動・実際挙動）
2. 新しい Issue を作成、ラベルは `bug`
3. 担当者を決めてアサイン
4. 修正 PR を出す（PR 本文に Issue 番号）
5. 再現手順で再テスト → 通れば close

> 再現できないバグは再現手順を詰めるところから始める

# テストを記録する

- 手動テストでも「何をテストしたか」を残す
- テスト項目表をリポジトリに置くと透明性が上がる

| 項目 | 期待 | 実際 | 日付 |
|------|------|------|------|
| 投稿作成（正常） | 一覧に表示 | OK | 6/22 |
| 投稿作成（空） | エラー表示 | OK | 6/22 |
| 投稿削除 | 一覧から消える | NG | 6/22 |

# 残タスクの見える化

- GitHub Issues を **オープン/クローズ** で絞って残りを可視化
- マイルストーン進捗バーで週の到達度を共有
- PR のドラフト/レビュー待ち/承認済みも把握
- 残り日数 × 残タスク数で現実性を判断

> 無理なら「提出物の最低ライン」と「欲しい機能」を区別する
