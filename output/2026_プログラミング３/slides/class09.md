# 要件定義とは

- **誰のために / 何を / なぜ作るか** を言語化する作業
- 設計・実装の前に「作るものを揃える」段階
- 合意なく作り始めると、後半で手戻りが発生する

> 要件が曖昧 → 設計が曖昧 → 実装が割れる → 統合で破綻

class: flex-layout natural-height

# 要件定義書の構成例

| 項目 | 内容 |
|------|------|
| 背景・目的 | なぜ作るか |
| ユーザー像 | 誰が使うか（ペルソナ） |
| ユースケース | 主な利用シーン |
| 機能要件 | 提供する機能の一覧 |
| 非機能要件 | 性能・セキュリティ・運用 |
| 対象外 | やらないこと（スコープ外） |

# 機能要件と非機能要件

**機能要件**（何ができるか）:
- ユーザーはログインできる
- レシピを投稿・検索できる
- コメントを書ける

**非機能要件**（どう提供するか）:
- 同時接続 100 人まで耐える
- レスポンス 2 秒以内
- HTTPS 必須、パスワードはハッシュ化

> 非機能を書かないと「動いたけど使い物にならない」が起きる

# 要件から Issue への落とし込み

1. 機能要件を **ユーザー価値単位** で分解
2. 各項目を Issue 化（タイトル = 動詞+対象）
3. ラベル付け（feature / doc / infra 等）
4. マイルストーン（週次）で束ねる
5. 担当者アサイン

> 1 Issue = 1 PR で終わらせられる粒度が理想

# KPT による振り返り

| 観点 | 意味 |
|------|------|
| Keep | うまくいった・継続したいこと |
| Problem | うまくいかなかった・課題 |
| Try | 次に試すこと |

- 毎週提出（電子フォーム、1点×回数）
- **「特になし」は 0 点** — 具体的に書く
- 個人の学びとチームの学びの両方を記録
