# チーム編成の考え方

- 3〜4名を基本（全員が **PR 1本以上** をマージ可能な規模）
- 役割は固定せず、**複数ロールを兼務** する
  - 例：実装＋レビュー、設計＋運用記録
- 得意領域が偏るより、**全員がコードとIssueに関わる**
- リーダーは工程管理・意思決定ログ担当を兼ねる
- 貢献の偏りはルーブリック上の減点対象

---

class: flex-layout natural-height

# 役割の例

| 役割 | 主な責務 |
|------|---------|
| リーダー | 全体進行・意思決定ログ |
| 実装担当 | 機能開発・PR作成 |
| 設計担当 | 仕様書の要件・基本設計セクション |
| レビュー担当 | PR確認・コードレビュー |
| 運用担当 | ブランチ戦略遵守・Issue整理 |

担当は重複してよいが、**全員がコードかIssueに関与** する。

# テーマ選定の基準

- **範囲が見切れる**：14週で完成する規模
- **分担可能**：3〜4人で並行作業できる構造
- **Issue が10件以上** 出せる内容の厚み
- 使用技術の習熟に時間をかけすぎない
- 既存OSSの単純模倣ではなく、**チーム固有の判断** が入る余地

---

<!--
class: flex-layout natural-height
-->

# テーマの例（参考）

| テーマ | 分担の取りやすさ |
|-------|---------------|
| 学内向け予約管理Web | 画面/DB/API で分割しやすい |
| コンソールゲーム | ロジック/UI/データで分割 |
| 簡易タスク管理アプリ | CRUD単位で分割 |
| 既存APIのクライアント | 機能ごとにPRを分割 |

**実装の大きさ** より **分担のしやすさ** を優先して選ぶ。

# リポジトリ作成チェックリスト

- [ ] チーム専用のリポジトリを作成（Public / Private は合意して決定）
- [ ] 全メンバーを **Collaborator** として招待
- [ ] `README.md` に以下を記載
  - テーマ・メンバー一覧・担当予定
  - 起動方法（書ける範囲で）
- [ ] `.gitignore` を追加
- [ ] ブランチ戦略を明文化（README または `docs/`）
- [ ] ライセンスは必要に応じて付与

---

<!--
class: flex-layout natural-height
-->

# ブランチ戦略の合意ポイント

- 基本は **GitHub Flow**（main + feature）を推奨
- 決めておく事項
  - ブランチ命名規則（例：`feature/issue-12-login`）
  - PRレビュアー数（最低1名は必須）
  - マージ方式（squash / merge / rebase）
  - 直接main pushの可否（原則禁止）
- **決定をドキュメント化** して全員で合意する
