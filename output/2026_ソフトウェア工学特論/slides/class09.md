# 機能要件と非機能要件

| 区分 | 内容 | 例 |
|------|------|---|
| 機能要件 | システムが **何をする** か | 「ログインできる」「予約一覧を表示」 |
| 非機能要件 | **どう振る舞う** か | 応答時間・利用環境・セキュリティ |

- 両方を書かないと後工程で齟齬が出る
- 非機能は **数値化** する（例：3秒以内、同時10人）
- まず機能要件を列挙、その後に制約で非機能を整理する

# 要件定義書のひな型

```markdown
## システム概要
〇〇を管理するWebアプリ

## 機能要件
- F-01 ユーザー登録
- F-02 ログイン
- F-03 予約作成
- F-04 予約一覧表示

## 非機能要件
- 応答時間：1操作あたり2秒以内
- 対象ブラウザ：Chrome 最新版
- 同時接続数：5ユーザー
```

**IDを振る** と後でIssue・設計書から参照しやすい。

# 要件 → Issue の粒度

- 要件ID1つに対して **Issueを1〜数件** 作成
- 1 Issue = 1 PRでマージできる単位が目安
- 大きすぎる場合は **サブタスクに分解**
- 小さすぎる場合は関連要件をまとめる
- ラベル例：`feature` / `bug` / `docs` / `design`

# Issueテンプレート例

```markdown
## 対応要件
F-03 予約作成

## 完了条件
- [ ] 予約登録フォーム画面
- [ ] 入力バリデーション
- [ ] DB保存処理
- [ ] 成功メッセージ表示

## 担当
@user1
```

チェックリスト式の **完了条件** がPRレビュー時の判定基準になる。

# KPTとは

| 項目 | 意味 |
|------|------|
| Keep | 続けたいこと・うまくいったこと |
| Problem | 問題・困ったこと |
| Try | 次に試したい改善 |

- 第9〜14週、**毎週末に1回** 提出
- 評価：1点×回数（提出漏れ・空白・意味のない文章は0点）
- 第15週のレポート③の材料になる

---

<!--
class: flex-layout natural-height

# KPT提出フォーマット

```markdown
## 第9週 KPT
### Keep
- Issueを要件に紐付けて作れた

### Problem
- 要件と非機能の切り分けで迷った

### Try
- 次週は非機能要件を数値化してから書く
```

**具体的な事象** を書くことが第15週のレポート③で効いてくる。
