## 課題の目的

チーム開発期間（第8〜14週）で自分が体験したプロジェクトを KPT（Keep / Problem / Try）の枠組みで振り返り、**次回のチーム開発に向けた具体的な改善提案** をまとめる。到達目標「プロジェクト管理手法に基づいて、チーム開発プロジェクトを管理・改善できる」に対応し、単なる感想ではなく、事象を次に活かすための提案に落とす力を養う。

## 課題の内容

### テーマ

**チーム開発プロジェクトの振り返りと改善提案レポート**

チーム開発期間で自身が経験したプロジェクトを対象に、KPT形式で振り返り、**実行可能な改善提案** を複数提示する。提案は精神論ではなく、次の開発で具体的に行動に移せるレベルまで落とし込む。

### 要件

**必須要件**

- 以下のセクションをすべて含めること
  - プロジェクトの概要（テーマ・チーム構成・期間・成果物）
  - 自分の役割と担当箇所
  - Keep（うまくいったこと・継続したいこと）: **3項目以上**
  - Problem（問題・課題として認識したこと）: **3項目以上**
  - Try（次に試したい改善提案）: **3項目以上**
  - 改善提案の優先度と実行計画（次回の何週目に何をするか）
  - 全体の振り返り（200字以上の散文）
- 各 Keep / Problem / Try には **具体的な事象・場面** を必ず添える（例: 「第11週で Issue の粒度が大きすぎて PR レビューが滞った」等）
- Try は **動詞で始まる行動レベル** の記述にすること（例: 「Issue は1PRでマージできる粒度まで分解する」）
- 毎週提出した KPT 自己評価を根拠として参照すること（引用・要約可）
- 文字数: 2500〜4500字（見出し・引用を除く）
- 生成AI を利用した場合は、投入したプロンプトをレポート末尾の付録に記載すること（`## 付録: 生成AI活用記録`）

**推奨要件**

- Problem と Try を対応付ける表を設ける
- チームメンバーのフィードバック・相互評価を引用する
- 自分の成長指標（定量的に測れるもの）を設ける

### 取り組みの手順

1. これまでに提出した自分の KPT 自己評価（第9〜14週分）を読み返し、事象を時系列で整理する
2. 事象を Keep / Problem に振り分ける。重複・上位概念化できるものはまとめる
3. Problem に対して Try（改善案）を複数考案する。実行可能性（次回着手できるか）で選別する
4. Try に優先度を付け、次回開発でいつ着手するかのタイムラインを描く
5. 全体を散文で振り返り、自分の成長と次への課題を言語化する
6. Markdown で構造化して整形する
7. 生成AI を活用した場合はプロンプト記録を付録に追記する
