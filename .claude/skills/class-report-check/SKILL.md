# レポート採点スキル

学生のレポート・成果物の提出物をルーブリックに基づいて自動採点し、観点別評価・フィードバックコメント・成績一覧CSVを出力するスキル。

## なぜこのスキルが必要か

レポートの採点は教員にとって最も時間のかかる業務の一つである。ルーブリックが明確に定義されている場合、採点基準は構造化されており、提出物をルーブリックの各段階と照合する作業は自動化の余地が大きい。アンカー採点方式（少数のサンプルで基準を確立してから一括採点）により、一貫した品質の自動採点を実現する。

## 前提条件

- 課題の出題文とルーブリックが存在すること（`/class-report` スキルで事前に作成、または手動で提供）
- 学生の提出物がディレクトリにまとめられていること
- 対応する提出物形式: Markdown (.md)、ソースコード (.py, .js, .c, .java 等)、PDF (.pdf)
- 画像単体は採点対象外（レポートやコード内に画像が含まれる場合は文脈として考慮する）

## デフォルト設定

| 項目 | 設定値 |
|------|-------|
| 配点 | 100点満点（`specs/rubric-schema.md` § 1.2） |
| ルーブリック段階・点数換算 | `specs/rubric-schema.md` § 2 を参照 |
| CSV文字コード | UTF-8 BOM付き（Excel対応） |

ユーザーが明示的に変更しない限り、上記のデフォルト値を適用する。点数換算デフォルト（理想的=80〜100%/標準点90%、標準的=60〜79%/標準点70%、未到達=0〜59%/標準点30%）は `specs/rubric-schema.md` § 2 を正本とする。

## 対話ルール

ユーザーへの質問・プレビュー・修正ループの作法は `specs/dialog-conventions.md` を正本とする。本スキルもこの規約に従う。

## 入出力契約

- **採点の正本**: 採点は `[レポート名]_課題.md` および `[レポート名]_ルーブリック.md` を**正本**として扱う。シラバスは参照しない。シラバスとのズレ（提出期限変更・観点改訂等）が気になる場合は `/class-report` を再実行して課題MD・ルーブリックMDを別名保存（`_v2`）し直すこと
- **入力ファイル形式**: `[レポート名]_課題.md` および `[レポート名]_ルーブリック.md` のスキーマは `specs/report-task-md-schema.md` を正本とする
- **出力ファイルパス**: `specs/output-layout.md` に従う（`output/[YYYY]_[教科名]/reports/[レポート名]/feedback/`）
- **ルーブリック構造・点数換算**: `specs/rubric-schema.md` を正本とする（点数換算デフォルトは § 2 に記載）
- **履修者名簿・チーム構成CSV**: `specs/roster-team-csv.md` を正本とする

## 処理フロー

```
Step 0: 採点対象の課題を選択（output/ 配下をスキャン）
Step 1: 課題内容・ルーブリック・メタデータの自動導出
Step 2: ルーブリックの確認・配点確定
Step 3: 提出物の対応表推定・受理
Step 4: アンカー採点（3件サンプリングで基準確立）
Step 5: 一括採点（残り全件）
Step 6: 採点結果レビュー
Step 7: ファイル出力（フィードバックMD + 採点一覧MD + 成績CSV）
Step 8: 完了レポート
```

### Step 0: 採点対象の課題を選択

`output/*/reports/*/*_課題.md` のパターンで課題ファイルを自動スキャンする。ヒット数によって分岐:

- **0件**: `/class-report` スキルでの事前作成を案内する。もしくはプレーンテキストで課題Markdownのパスを直接受理する（フォールバック）
- **1件**: 「この課題で採点しますか？」とプレーンテキストで確認する
- **複数件**: `AskUserQuestion` で採点対象を1件選択させる（例: 「2026_コンパイラ / レポート1」「2026_アルゴリズム / レポート2」）

選択された課題Markdownのパスから、以下を自動導出する:

- **年度**: `output/[年度]_[教科名]/...` の `[年度]` 部分
- **教科名**: 同上の `[教科名]` 部分
- **レポート名**: `reports/[レポート名]/...` の `[レポート名]` 部分
- **出力先ディレクトリ**: `output/[年度]_[教科名]/reports/[レポート名]/`

パスから年度・教科名が取れない（例: `output/` 外のパスがフォールバック入力された）場合のみ、プレーンテキストで個別に聞く。

### Step 1: 課題内容・ルーブリック・メタデータの読み込み

Step 0 で特定した出力先ディレクトリから以下を自動検出・読み込みする:

- **課題Markdown**: Step 0 で選択済み
- **ルーブリックMarkdown**: 同階層の `[レポート名]_ルーブリック.md` を自動検出。見つからない場合はプレーンテキストでルーブリックMDのパスをユーザーに直接聞く（課題MDの「評価基準」テーブルは観点名・配点・概要のみ保持し、各段階の記述は持たないため、課題MD単体では採点不可）
- **提出物ディレクトリ**: 同階層の `submissions/`（Step 3 で使用）

課題Markdownの `## 基本情報` テーブルから以下のメタデータを抽出する:

| 表示項目 | 課題文の抽出元セクション |
|---------|----------------------|
| 科目名 | `## 基本情報` テーブルの「科目名」行 |
| レポート名 | `## 基本情報` テーブルの「レポート名」行 |
| 提出形態（個人/チーム） | `## 基本情報` テーブルの「提出形態」行 |
| 課題の概要 | `## 課題の目的` セクション |
| 提出物の要件 | `## 提出物の仕様` セクション |
| 評価観点 | `## 評価基準` テーブル |

`## 基本情報` テーブルが存在しない（手動作成された課題）場合は、提出形態のみ `AskUserQuestion` で確認する:
- options: 「個人提出」「チーム提出（チーム共通点数）」

**チーム提出の場合**: 「チーム構成は Step 3 で履修者名簿と一緒に確認します」と予告だけ伝え、入力は Step 3 にまとめる（同時に聞いた方が教員が必要なファイルをまとめて準備でき、入力ポイントが1箇所に集約されるため）。

読み込んだ内容をサマリー表示する:

```
=== 課題情報 ===
科目名: [科目名]
レポート名: [レポート名]
提出形態: [個人 / チーム]
課題の概要: [概要]
提出物の要件: [要件]

=== ルーブリック ===
| 評価観点 | 理想的 | 標準的 | 未到達 | 配点 |
|----------|--------|--------|--------|------|
| ... | ... | ... | ... | ... |
| 合計 | — | — | — | 100点 |

=== 点数換算基準（ルーブリックファイルから読み取り）===
| 段階 | 点数範囲 | 標準点 |
|------|---------|--------|
| ... | ... | ... |
```

ルーブリックファイルに「点数換算基準」セクションが含まれている場合、その値を読み取ってデフォルト設定より優先して使用する。

ルーブリックファイルの「採点メモ欄」セクションは手動採点用のテンプレートであり、自動採点では無視する。

### Step 2: ルーブリックの確認・配点確定

`AskUserQuestion` でルーブリックを確認する:
- options: 「このルーブリックで採点を進める」「ルーブリックを修正・補完したい」

**配点の確定**: ルーブリックに配点が含まれていない場合、`AskUserQuestion` で方式を確認する:
- options: 「全観点均等配点（自動計算）」「観点ごとに配点を設定する」

均等配点の場合は合計点（デフォルト100点）から各観点の配点を自動計算する。

**点数換算基準の確認**:
```
=== 点数換算基準 ===
| 段階 | 点数範囲 | 標準点 |
|------|---------|--------|
| 理想的 | 80〜100% | 90% |
| 標準的 | 60〜79% | 70% |
| 未到達 | 0〜59% | 30% |
```

`AskUserQuestion` で確認する:
- options: 「この基準で進める」「基準を変更したい」

### Step 3: 提出物の対応表推定・受理

**学生情報の収集**: 提出物の照合に必要な情報をまとめて収集する。チーム提出の場合は両方、個人提出の場合は履修者名簿のみを聞く。

履修者名簿（個人・チーム共通、オプション）:
```
履修者名簿のファイル（CSV等）があれば、パスを教えてください
（ない場合は「なし」と入力してください。名簿がない場合は提出物から検出された学生のみを採点対象とします）:
```

チーム構成（チーム提出のみ、必須）:
```
チーム構成を教えてください。以下のいずれかの方法で入力できます:
- チーム構成CSVのファイルパス
- チーム名とメンバーの一覧を直接入力
```

ファイル形式・カラム名は `specs/roster-team-csv.md` に従う。名簿が提供された場合は、名簿に含まれる学生のうち提出物が検出されなかった学生を「未提出」として記録する。名簿がない場合は、提出物として検出されたもののみを採点対象とし、未提出判定は行わない。

出力先ディレクトリ内の `submissions/` フォルダを自動検出し、そこを走査対象とする。見つからない場合はプレーンテキストで走査対象のパスを聞く。

`references/batch-processing-guide.md` の「提出物の対応表推定」に従い、以下を行う:

1. submissions/ 配下を再帰的に走査
2. 学籍番号正規表現（デフォルト `\d{7}`）、フォルダ名、ファイル名、名簿との照合で学生-提出物対応を推定
3. 推定結果を表形式で表示

```
=== 提出物対応表（推定）===
| # | 学籍番号 | 氏名 | 検出されたファイル | 検出元 |
|---|---------|------|-------------------|-------|
| 1 | 2450001 | 山田太郎 | report.md, main.py | フォルダ名 |
| 2 | 2450002 | 佐藤花子 | 佐藤花子_report.pdf | ファイル名 |
| ? | ?       | ?     | misc.txt          | 識別不可 |

ファイル形式の内訳:
  - Markdown (.md): [N]件
  - コード: [N]件
  - PDF (.pdf): [N]件
  - 採点保留（画像単体・非対応形式・101ページ超PDF・読み込みエラー）: [N]件
  - 無視（システムファイル等）: [N]件

未提出（名簿あり時のみ）: [N]名
```

`AskUserQuestion` で確認する:
- options: 「この対応表で採点を開始する」「学生識別子（正規表現）を調整したい」「対応表を手動で修正したい」

「識別不可」の行がある場合、採点開始を許可せず、上記いずれかの修正オプションで解消してから次に進む。

### Step 4: アンカー採点（基準確立）

採点の一貫性を確保するため、全件採点の前にサンプル3件で基準を確立する。この方式により「同じ品質のレポートには同じ点数」を実現する。

`references/anchor-review-guide.md` に従い、提出物の記述量（ファイル形式別に定義された測定方法で算出）で3件をサンプリングする:
1. **記述量が最大の1件**（高品質の可能性が高い）
2. **記述量が最小の1件**（低品質の可能性が高い）
3. **記述量が中央値付近の1件**

提出件数が3件未満の場合は、全件を直接詳細採点する（Step 4 と Step 5 を統合する）。詳細は同ガイド参照。

各サンプルを `references/scoring-guide.md` に従って採点し、結果を表示する:

```
=== アンカー採点: [学籍番号] [氏名] ===

### [観点1: 内容の正確さ・深さ]
評価: 標準的
点数: [N]点 / [M]点
根拠: ルーブリックの「標準的」の定義「〜〜〜」に合致。
      具体的には、提出物の〜〜〜の部分で〜〜〜を達成しているが、
      〜〜〜については不足している。

### [観点2: 構成・論理展開]
...

---
総合点: [N]点 / 100点
```

3件のアンカー採点後、`AskUserQuestion` で確認する:
- options: 「この採点基準で一括採点を開始する」「採点結果を修正したい」「別の3件でやり直したい」

「採点結果を修正したい」場合はプレーンテキストで「どの提出者のどの観点を修正しますか？」と質問し、修正を反映する。

アンカー採点が確定した時点で、以降の一括採点ではこの3件を基準として参照する。

### Step 5: 一括採点

Step 4 で確定したアンカーを基準に、残り全件を採点する。

各提出物の処理:
1. ファイルを読み込む（`references/batch-processing-guide.md` のファイルタイプ別読み込みルールに従う）
2. `references/scoring-guide.md` に従い、アンカーとの相対比較を含めて各観点を評価する
3. `references/feedback-guide.md` に従い、フィードバックコメントを生成する

採点中の進捗をユーザーに報告する:
```
採点進捗: [N]/[M] 完了 — 現在: [学籍番号] [氏名]
```

**一貫性チェック（採点完了後に実施）**:
- 点数分布に極端な偏り（全員満点、全員最低点等）がないか確認する
- アンカーと同じ段階の評価を受けた提出物の根拠が一貫しているか確認する
- 問題があれば修正してから Step 6 に進む

### Step 6: 採点結果レビュー

全件の結果をサマリー表示する:

```
=== 採点完了サマリー ===
提出者数: [N]名 / 採点完了: [N]名 / 採点保留: [N]名 / 未提出: [N]名

採点保留がある場合（教員の手動採点が必要）:
| 学籍番号 | 氏名 | 理由 |
|---------|------|------|
| 2450001 | 山田太郎 | 画像単体のため採点保留・手動採点要 |
| ... | ... | ... |

点数分布:
  90〜100点: [N]名 ████████
  80〜 89点: [N]名 ██████
  70〜 79点: [N]名 ████
  60〜 69点: [N]名 ██
   0〜 59点: [N]名 █

平均点: [N]点 / 最高点: [N]点 / 最低点: [N]点 / 中央値: [N]点

観点別 評価分布:
| 評価観点 | 理想的 | 標準的 | 未到達 | 平均点 |
|----------|--------|--------|--------|--------|
| [観点1] | [N]名 | [N]名 | [N]名 | [N]点 |
| [観点2] | [N]名 | [N]名 | [N]名 | [N]点 |
| ... | ... | ... | ... | ... |
```

`AskUserQuestion` で確認する:
- options: 「この結果でファイル出力に進む」「特定の学生の採点を修正したい」「全体の採点傾向を調整したい」

「特定の学生の採点を修正したい」: プレーンテキストで学籍番号と修正内容を聞く。期限後に提出された学生を後から1〜2名追加採点したい場合や、特定学生の点数だけ修正したい場合もこの選択肢で対応する（Step 7 でファイル出力した後に再度本スキルを起動してこのフローに入ってよい）。
「全体の採点傾向を調整したい」: プレーンテキストで調整内容を聞き（例: 全体+5点シフト）、`references/scoring-guide.md` の調整ルールに従い反映する。

修正後は再度サマリーを表示し、確定するまで繰り返す。

### Step 7: ファイル出力

`AskUserQuestion` で出力形式を確認する:
- options: 「3種類すべて出力（推奨）」「個別フィードバックのみ」「成績一覧CSVのみ」

出力先:
```bash
mkdir -p "output/[年度]_[教科名]/reports/[レポート名]/feedback"
```

**既存判定**: 採点結果は再採点で更新されることがあるため、`feedback/` ディレクトリ配下のファイルおよび `[レポート名]_採点一覧.md`・`[レポート名]_成績.csv` のいずれかが既に存在する場合は `specs/file-output-policy.md` § 2 に従って確認する。本スキルは「`feedback/` を含む採点成果物一式」を1単位として扱い、別名保存時は `feedback_v2/` ディレクトリと `[レポート名]_採点一覧_v2.md` / `[レポート名]_成績_v2.csv` のように同期して `_v2` を付ける（同じ採点回のセットを後から識別できるようにするため）。レポート出題物（`[レポート名]_課題.md` 等）は採点とは別文脈なので既存判定の対象外。

出力するファイル:

**1. 個別フィードバックMarkdown**（提出者ごと）
`output/.../reports/[レポート名]/feedback/[学籍番号]_[氏名]_feedback.md`

`references/output-format.md` のフォーマットに従う。内容:
- 観点別評価（段階 + 点数 + 採点根拠）
- 総合点
- フィードバックコメント（良い点・改善点・次のアクション）

**2. 採点一覧Markdown**（教員確認用）
`output/.../reports/[レポート名]/[レポート名]_採点一覧.md`

全提出者の採点結果を表形式で一覧表示。

**3. 成績一覧CSV**（成績処理用）
`output/.../reports/[レポート名]/[レポート名]_成績.csv`

`references/csv-format.md` のフォーマットに従う。UTF-8 BOM付き。チーム提出の場合はメンバー全員に同一点数を展開する。

### Step 8: 完了レポート

```
=== レポート採点完了 ===
科目名: [科目名]
レポート名: [レポート名]
採点日: [YYYY-MM-DD]

採点結果:
  採点済み: [N]名 / 採点保留: [N]名 / 未提出: [N]名
  平均点: [N]点 / 最高点: [N]点 / 最低点: [N]点

採点保留（教員の手動採点要）: [N]名
  - [学籍番号] [氏名]: [理由]
  - ...

出力先: output/[YYYY]_[教科名]/reports/[レポート名]/

生成ファイル:
- 個別フィードバック: feedback/ 配下 × [N]件
- 採点一覧: [レポート名]_採点一覧.md
- 成績CSV: [レポート名]_成績.csv
```

## 採点品質の担保

このスキルは自動採点としてそのまま使用されることを前提としている。品質を担保するため、以下の仕組みを組み込んでいる:

1. **アンカー採点方式**: 3件のサンプルで基準を確立し、ユーザーが承認してから一括採点に進む。これにより「教員が納得した基準」で全件を採点する
2. **ルーブリック厳密準拠**: 各観点の評価は必ずルーブリックの定義を引用し、どの段階に合致するかを明示する
3. **採点根拠の明示**: 点数だけでなく、なぜその評価になったかの根拠を全件記録する
4. **一貫性チェック**: 全件採点後に分布チェック・アンカー整合チェックを自動実施する
5. **教員レビューステップ**: Step 6 で全体サマリーを提示し、修正の機会を設ける

## 対応する提出物形式

| 形式 | 拡張子 | 読み込み方法 |
|------|--------|-------------|
| Markdown | .md, .txt | Read ツールで直接読み込み |
| ソースコード | .py, .js, .c, .java, .rb 等 | Read ツールで直接読み込み |
| PDF | .pdf | Read ツールで読み込み（ページ数別のルールは `batch-processing-guide.md` 参照） |

画像単体・非対応形式のみ・PDF 101ページ超・読み込みエラーは「採点保留」として扱い、成績CSVの点数列は空欄、備考列に理由を記録する。未提出とは明確に区別する（詳細は `references/csv-format.md` および `references/batch-processing-guide.md` 参照）。レポートやコード内に埋め込まれた画像は文脈として考慮する。
