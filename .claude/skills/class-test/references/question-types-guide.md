# 問題形式別 作問ガイド

Moodle 4.x の自動採点対応形式ごとの作問ルール・品質基準。

## 1. 多肢選択 (multichoice)

### 用途
- 概念理解・判断力の確認
- 基礎〜応用まで幅広い難易度に対応

### 作問ルール

**選択肢数**: 4択を標準とする（3択は容易すぎ、5択以上は解答時間が延びる）

**正解**:
- 単一選択（ラジオボタン）を基本とする
- 複数選択（チェックボックス）は「正しいものをすべて選べ」形式で使用。不正解選択肢には負の fraction を設定すること

**ディストラクタ（誤答選択肢）の設計**:
- もっともらしい誤答を作る（ランダムな誤答は避ける）
- 典型的な誤解・よくある間違いをディストラクタにする
- 「上記のいずれでもない」「すべて正しい」は原則使用しない
- 選択肢の長さをできるだけ揃える（正解だけ長い/短いのは避ける）

**問題文**:
- 否定形（「〜でないものはどれか」）は極力避ける。使う場合は「**ない**」を強調する
- 問題文だけで何を問われているか明確にわかること

**フィードバック**:
- 各選択肢に個別フィードバックを設定する
- 正解: なぜ正しいかを簡潔に説明
- 不正解: なぜ間違いかを簡潔に説明（学習機会にする）

### 品質チェックリスト
- [ ] 4択になっているか
- [ ] 正解が1つに定まるか（単一選択の場合）
- [ ] ディストラクタがもっともらしいか
- [ ] 選択肢の長さが揃っているか
- [ ] 問題文が明確か
- [ ] 各選択肢にフィードバックがあるか

---

## 2. 数値問題 (numerical)

### 用途
- 計算量の算出（O記法の具体値）
- アルゴリズムのトレース結果
- 数式の計算結果

### 作問ルール

**許容誤差の設定**:
- 整数解が期待される場合: tolerance = 0
- 小数解の場合: 問題文で精度を指定し（「小数第2位まで」）、tolerance をそれに合わせる
- 計算途中の丸め誤差を考慮して適切な tolerance を設定する

**複数正解の設定**:
- 複数の正解値がありうる場合（例: 等価な表現）は、複数の `<answer>` で対応
- 部分点を設定する場合は fraction で配分

**単位**:
- 原則として単位は問わない（問題文で「〜の値を答えよ」と限定する）
- 必要な場合は `<units>` で設定

**問題文**:
- 求める値を明確に指定する
- 「整数で答えよ」「小数第N位まで答えよ」など精度を指示する
- トレース問題では入力値・初期状態を明示する

### 品質チェックリスト
- [ ] 正解値が一意に定まるか
- [ ] 許容誤差が適切か
- [ ] 精度の指示が問題文にあるか
- [ ] 計算過程に曖昧さがないか

---

## 3. 穴埋め Cloze (cloze / multianswer)

### 用途
- コードの穴埋め
- 複数概念の組み合わせ確認
- 文中の複数箇所を同時に問う

### 作問ルール

**Cloze構文**:
```
{配点:タイプ:=正解#フィードバック~%部分点%別解#フィードバック~誤答#フィードバック}
```

**サブ問題タイプの使い分け**:
| タイプ | 用途 |
|---|---|
| SHORTANSWER (SA) | キーワード・関数名・変数名の穴埋め |
| NUMERICAL (NM) | 数値の穴埋め |
| MULTICHOICE (MC) | ドロップダウンで選択（文脈中の選択） |

**コード穴埋めの注意点**:
- コードブロックは `<pre><code>` タグで囲む
- 穴埋め箇所は文脈から推測可能だが一意に定まるようにする
- 大文字小文字を区別する場合は `SHORTANSWER_C` (SAC) を使用
- プログラミング言語の構文に合わせた正解バリエーションを設定する（例: `==` と `=` の違い）

**配点**:
- `<defaultgrade>` はサブ問題の配点合計と一致させる
- 各穴埋め箇所の配点は `{配点:タイプ:...}` の配点で設定

**フィードバック**:
- 各穴埋め箇所の正解・誤答にフィードバックを設定する

### 品質チェックリスト
- [ ] 穴埋め箇所が文脈から一意に定まるか
- [ ] 正解のバリエーション（表記揺れ）が網羅されているか
- [ ] コードの穴埋めは構文的に正しいか
- [ ] defaultgrade とサブ問題の配点合計が一致するか

---

## 4. マッチング (matching)

### 用途
- 概念と定義の対応付け
- アルゴリズム名と計算量の対応
- データ構造と操作の特徴の対応

### 作問ルール

**ペア数**: 4〜6ペアを標準とする（3ペア以上が必須）

**ダミー選択肢**:
- 1〜2個のダミー（マッチしない余分な選択肢）を追加して難易度を調整する
- ダミーは `<subquestion>` の `<text>` を空にして `<answer>` のみ設定する

**ペアの設計**:
- 同じカテゴリの概念でペアを構成する（例: すべてソートアルゴリズム名と計算量）
- 異なるカテゴリを混ぜない（マッチングの意味がなくなる）
- 答え側の選択肢が一意にマッチするようにする

**フィードバック**:
- 個別ペアへのフィードバックはMoodleでサポートされないため、`<generalfeedback>` で全ペアの正解を説明する

### 品質チェックリスト
- [ ] 4ペア以上あるか
- [ ] ダミー選択肢があるか（難易度調整）
- [ ] ペアが一意にマッチするか
- [ ] 同カテゴリの概念で構成されているか

---

## 5. 記述問題 Short answer (shortanswer)

### 用途
- 用語・記号の正確な回答
- 略称・正式名称の確認
- 短い数式や記号の記述

### 作問ルール

**正解のバリエーション**:
- 表記揺れを網羅する（全角/半角、大文字/小文字、略称/正式名称）
- `<usecase>0</usecase>` で大文字小文字を区別しないのが安全
- ワイルドカード `*` を活用する（例: `*スタック*` で「スタック」を含む回答にマッチ）
- 部分点を設定する場合は複数の `<answer>` で fraction を配分

**catch-all 回答**:
- `<answer fraction="0">` に `*` を設定し、不正解時のフィードバックを表示する

**問題文**:
- 求める回答の形式を明示する（「日本語で答えよ」「英語で答えよ」「記号で答えよ」）
- 回答の長さの目安を示す（「一語で答えよ」）

### 品質チェックリスト
- [ ] 正解が一意に定まるか
- [ ] 表記揺れのバリエーションが網羅されているか
- [ ] 回答形式の指示が問題文にあるか
- [ ] catch-all フィードバックがあるか

---

## 6. 計算問題 Calculated (calculated)

### 用途
- 値がランダム化される計算問題（AI耐性が高い）
- 計算量の具体的な算出
- 公式の適用

### 作問ルール

**変数の設計**:
- 変数名は `{a}`, `{b}`, `{n}` など短くわかりやすい名前を使う
- 変数の範囲（minimum/maximum）は現実的な値にする
- 小数点桁数（decimals）は問題の性質に合わせる（整数なら 0）
- 分布は uniform（一様分布）を標準とする

**計算式**:
- Moodleの数式関数を使用: `pow()`, `log()`, `sqrt()`, `abs()`, `ceil()`, `floor()`, `round()`, `pi()` 等
- 除算の場合はゼロ除算にならないよう変数範囲を設定する

**データセット**:
- 最低10セットのデータセット値を生成する
- 各セットで正解値が現実的な範囲に収まることを確認する
- edge caseを含める（最小値、最大値付近）

**許容誤差**:
- 整数解の場合: tolerance = 0.01（丸め誤差対策）
- 小数解の場合: 問題で指定した精度に合わせた tolerance

**AI耐性のポイント**:
- 具体的な数値を使った計算を求める（一般論では答えられない）
- 変数がランダム化されるため、過去の解答を暗記しても意味がない
- 計算過程を理解していないと正解できない

### 品質チェックリスト
- [ ] 変数範囲が現実的か
- [ ] ゼロ除算等の異常値が発生しないか
- [ ] 最低10セットのデータセット値があるか
- [ ] 許容誤差が適切か
- [ ] 計算式がMoodleの関数構文に準拠しているか

各形式の想定解答時間は `question-design-guide.md` の「形式別の想定解答時間」表を参照すること。問題構成設計時に時間バッファルールを満たすよう検証する。

## 共通の品質基準

### 問題文の品質
- 何を問われているか1文目で明確にわかること
- 必要な前提条件・情報がすべて問題文に含まれていること
- 曖昧な表現を避ける（「適切な」→具体的な基準を示す）
- 日本語として自然で読みやすいこと

### フィードバックの品質
- 正解の場合: なぜ正しいかを簡潔に説明（1〜2文）
- 不正解の場合: なぜ間違いかを説明し、正解に至るヒントを提示
- 学習者が次に何をすべきか（復習すべき内容）を示す

### AI耐性の原則
1. **具体的な値・手順を問う**: 一般的な説明ではなく、特定の入力に対する出力を求める
2. **トレースを求める**: アルゴリズムの実行過程を追わせる
3. **比較・判断を求める**: 複数の選択肢から最適なものを選ばせ、その理由を問う
4. **ランダム化**: Calculated問題で値を毎回変える
5. **時間制約**: 制限時間内に解答する前提の問題量にする
