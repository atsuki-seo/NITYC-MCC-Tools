# シラバス解析スキル

`class-syllabus` スキルが生成したシラバスMarkdownを読み込み、後続スキル（`class-test`・`class-report` など）が利用するための構造化情報を抽出するスキル。

## なぜこのスキルが必要か

シラバスからの情報抽出ロジック（基本情報・スケジュール・到達目標・ルーブリック・名前正規化など）は、テスト作成・レポート作成・成績処理など複数のスキルで共通して必要になる。各スキルに同じ抽出ロジックを持たせると二重メンテが発生するため、解析処理をこのスキルに一元化する。

## 前提条件

- シラバスMarkdownファイルが `output/` 配下に存在すること（`/class-syllabus` スキルで事前に作成）
- 出力はコンテキスト表示のみ。ファイル書き出しは行わない

## 対話ルール

ユーザーへの質問・プレビュー・修正ループの作法は `specs/dialog-conventions.md` を正本とする。本スキルもこの規約に従う。

## コンテキスト出力契約

本スキルは抽出結果をコンテキストに残し、後続スキル（`class-test` / `class-report` / `class-slides` 等）に引き渡す。コンテキストに残す情報の正本は `specs/syllabus-markdown-schema.md` § 9 を参照。Step 2〜4 の抽出は § 9.1 の項目をすべて満たすこと。

---

## 処理フロー

```
Step 0: シラバスMarkdownの特定
Step 1: ファイル読み込み
Step 2: 基本情報の抽出・表示
Step 3: スケジュール（レポート・試験）の抽出・表示
Step 3.5: 授業計画の抽出・表示
Step 4: 到達目標・ルーブリックの抽出・表示
Step 4.5: 概要・MCC対応・授業の属性の抽出（原文保持）
Step 5: 抽出結果全体の確認
```

すべての Step で **保持（コンテキスト）** と **表示（ユーザー向け）** を分離する。長大な項目（授業計画・概要・MCC対応・授業の属性・ルーブリック本文）はサマリーのみ表示し、原文はコンテキストに保持する。詳細は `specs/syllabus-markdown-schema.md` § 9.1.1 を参照。

### Step 0: シラバスMarkdownの特定

以下の優先順位でシラバスファイルを特定する:

1. **引数で渡された場合**: 引数のパスをそのまま使用する
2. **会話コンテキストに既にシラバスが読み込まれている場合**: `AskUserQuestion` で確認する
   - options: 「現在のコンテキストに読み込まれている『[科目名]』で進める」「別のシラバスを指定する」
3. **`output/` 配下にシラバスMarkdownが存在する場合**: `output/*/[年度]開講*.md` を glob でリストアップし、`AskUserQuestion` の options として提示する。3ファイル以下なら全件、それ以上なら代表的なものを出して "Other" で手動入力を受け付ける
4. **該当がなかった場合**: プレーンテキストで直接パスを質問する:
   ```
   シラバスMarkdownファイルのパスを教えてください
   （例: output/2026_アルゴリズム/26開講情報工学科3アルゴリズム.md）:
   ```

シラバスMarkdownが見つからない場合は `/class-syllabus` スキルでの事前作成を案内する。

### Step 1: ファイル読み込み

特定したファイルを Read ツールで読み込む。読み込み失敗時はエラー内容を提示し、Step 0 に戻る。

### Step 2: 基本情報の抽出・表示

`specs/syllabus-markdown-schema.md` § 2.2 の抽出仕様に従って以下を抽出する:

- 科目名
- 開講年度（`## 基本情報` 表 > ファイルパス `output/[YYYY]_*` の優先順位で取得）
- 担当教員
- 対象学年
- 開講期
- 単位
- 教科書/教材（URL または書名。文字列として透過保持）
- 開設学科
- 評価割合（評価方法ごとの配点比率）
  - **原文名**（Excel テンプレート準拠の評価方法名）と **カテゴリキー**（`試験` / `小テスト` / `レポート` / `発表` / `ポートフォリオ` / `その他`）の両方を保持する。マッピングルールは § 2.3 を参照

表示フォーマット:
```
=== シラバス基本情報 ===
科目名: [科目名]
開講年度: [YYYY]年度
担当教員: [教員名]
対象学年: [学年]
開講期: [期]
単位: [単位]
教科書/教材: [URL または書名]
開設学科: [学科]
評価割合:
  - [原文名1]: [N]%（カテゴリ: [カテゴリキー]）
  - [原文名2]: [N]%（カテゴリ: [カテゴリキー]）
  - ...
```

基本情報が見つからない項目は「（記載なし）」と表示する。エラーにはしない。

### Step 3: スケジュールの抽出・表示

レポート・試験のスケジュールを一括で抽出する。抽出ロジックの詳細はプロジェクトルートの `specs/syllabus-markdown-schema.md` を参照。

表示フォーマット:
```
=== レポート・課題スケジュール ===
| レポート名 | 提出週 | テーマ |
|-----------|--------|-------|
| レポート1 | 第8週 | ... |
| ... | ... | ... |

=== 試験スケジュール ===
| 試験名 | 実施週 | 範囲 |
|--------|--------|------|
| 小テスト1 | 第5週 | 第1〜4週 |
| 期末試験 | 第15週 | 第1〜14週 |
| ... | ... | ... |
```

スケジュールに該当する記載がない場合は「（該当なし）」と表示する。両方とも抽出できない場合でも処理は継続する（後続スキルで必要に応じて手動入力を求める）。

抽出結果に `（要確認）` が含まれる場合は、`specs/syllabus-markdown-schema.md` § 4.4 の伝播ルールに従い、Step 5 で警告表示とユーザー確定の促しを行う。

### Step 3.5: 授業計画の抽出・表示

`## 授業計画` テーブル全体をコンテキストに保持する（各週の授業内容・到達目標対応を含む）。保持値は後続スキル（`class-test` / `class-slides`）が週別の範囲設定・スライド生成で参照する。

表示はサマリーのみに留める（§ 9.1.1 の保持/表示分離ルール）:

```
=== 授業計画 ===
全[N]週分を抽出（授業内容・到達目標対応をコンテキストに保持）
先頭3週のサマリー:
  第1週: [授業内容の先頭30字程度]
  第2週: [授業内容の先頭30字程度]
  第3週: [授業内容の先頭30字程度]
```

ユーザーが「全週分を見たい」と指示した場合のみ全文展開する。授業計画テーブルが抽出できない場合は「（記載なし）」として処理を継続する（後続スキルで必要に応じて手動入力を求める）。

### Step 4: 到達目標・ルーブリックの抽出・表示

表示フォーマット:
```
=== 到達目標 ===
[シラバスの到達目標をそのまま表示]

=== ルーブリック ===
| 評価観点 | 理想的 | 標準的 | 未到達 |
|----------|--------|--------|--------|
| ... | ... | ... | ... |
```

ルーブリックの各記述が長大な場合は、表示時は先頭50字程度でトリミングして「...」を付けるサマリー表示としてよい。原文は必ずコンテキストに完全保持する（§ 9.1.1）。

### Step 4.5: 概要・MCC対応・授業の属性の抽出（原文保持）

以下の3項目を **Markdown 原文のまま** コンテキストに保持する（§ 9.1 参照）:

- 概要: `## 概要` セクションの本文テキスト（散文そのまま）
- MCC対応: `## MCC対応` セクションのテーブル原文（Markdown テーブル文字列）
- 授業の属性: `## 授業の属性` セクションのテーブル原文（Markdown テーブル文字列）

各項目のセクションが存在しない場合はそのカテゴリは「（記載なし）」として扱い、処理は継続する。

表示はサマリーのみ（§ 9.1.1）:

```
=== 概要 ===
[先頭100字程度]…（全文はコンテキストに保持）

=== MCC対応 ===
全[N]項目を抽出（テーブル原文をコンテキストに保持）

=== 授業の属性 ===
全[N]項目を抽出（テーブル原文をコンテキストに保持）
```

### Step 5: 抽出結果の確認

Step 2〜4.5 の全抽出結果（サマリー）をまとめて提示した後、`AskUserQuestion` で確認する:
- options: 「この内容で進む（後続スキルで使用する）」「抽出内容を修正したい」

「修正したい」が選ばれた場合、プレーンテキストで修正箇所を聞き、該当項目のみ差し替える。スケジュールの追加・削除もここで対応する。

**`（要確認）` の警告ガード**: 試験スケジュールに `（要確認）` の行が含まれる場合は、この Step で以下の警告を表示し、ユーザーに確定を促す（`specs/syllabus-markdown-schema.md` § 4.4 参照）:

```
⚠️ 試験範囲が未確定の項目があります:
  - [試験名]: [実施週] / 範囲: （要確認）
後続スキル（class-test 等）で再確認が必要です。ここで範囲を確定することもできます。
```

ユーザーがここで範囲を確定した場合は該当項目を差し替える。確定されなかった場合は `（要確認）` のままコンテキストに保持し、後続スキルでの再確認に委ねる。

確認完了後は、抽出結果がコンテキストに残っている状態で本スキルを終了する。後続スキル（`class-test`・`class-report` 等）はこの抽出結果を参照して動作する。

---

## 後続スキルからの呼び出し方

`class-test`・`class-report` などの後続スキルは、Step 0 で本スキルを呼ぶよう案内すること。呼び出し方の例:

```
このスキルは事前にシラバス情報を読み込む必要があります。
`/class-syllabus-parse` を実行してから、続きを進めてください。
```

本スキルを実行すると、以降のコンテキストに以下の情報が残る:
- シラバスの生Markdownテキスト
- Step 2〜4 で抽出・表示された構造化情報

後続スキルはこれらを再度 Read せず、コンテキストに残っている情報を参照する。
