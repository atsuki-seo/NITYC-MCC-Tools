# レポート課題設計ガイド

Step 3（テーマ確定）と Step 4（課題詳細設計）で参照する設計ルール。

## シラバス記述密度の判定基準

レポートごとに「授業の進め方」と「授業計画」の記述内容から判定する。

### 詳細あり

以下の条件を **2つ以上** 満たす場合:
- テーマ名が明記されている（例: 「字句解析の設計とプログラミング実践」）
- 提出物の形式・内容が記述されている（例: 「ソースコード + 動作検証用コード」）
- 取り組みの方向性や要件が記述されている（例: 「正規表現を用いた字句解析器を設計・実装する」）

**例**: コンパイラのシラバス「レポート①: 字句解析の設計とプログラミング実践（第8週提出）」→ テーマ名あり + 方向性あり = 詳細あり

### 概要のみ

以下のいずれか1つのみ該当する場合:
- テーマ名のみ（内容要件の記述なし）
- 提出週のみ（テーマ名の記述なし）
- 「レポート」の記載があるが具体的な内容は不明

**例**: 「第8週: レポート提出」→ 提出週のみ = 概要のみ

### 記述なし

以下の場合:
- 評価割合にレポートの配点があるが、「授業の進め方」「授業計画」にレポートの具体的な記述がない
- レポートへの言及が全くない

**例**: 評価割合でレポート30%だが、授業の進め方にレポートの記述なし = 記述なし

## テーマ候補生成ルール（記述なしの場合）

シラバスの到達目標・授業計画・ルーブリックから、以下の方針でテーマ候補を3〜5個生成する。

### 生成の方針

1. **到達目標ベース**: 各到達目標に対応するレポートテーマを1つ以上考える
2. **授業計画ベース**: 授業計画の単元区切り（3〜5週のまとまり）ごとにテーマを考える
3. **実践重視**: 知識の再現ではなく、応用・分析・創造を求めるテーマにする
4. **多様性**: 調査型・実装型・分析型・設計型のバリエーションを含める

### テーマの分類

| テーマ型 | 説明 | 適する科目例 |
|---------|------|-------------|
| 調査・比較型 | 複数の技術・手法を調査し比較分析する | プログラミング言語比較、アルゴリズム比較 |
| 実装・制作型 | プログラムや成果物を作成し報告する | コンパイラ、プログラミング演習 |
| 分析・考察型 | 事例やデータを分析し考察する | ソフトウェア工学、ネットワーク |
| 設計・提案型 | システムや手法を設計し提案する | チーム開発、プロジェクト管理 |

### 提示フォーマット

各テーマ候補には以下を含める:
- テーマ名（簡潔で具体的な名前）
- 1〜2行の説明（何をするか・何を学ぶか）
- テーマ型の分類

## 課題詳細設計ルール（Step 4）

### 必須項目

1. **課題の目的**（1〜2文）
   - 到達目標のどれに対応するかを明示する
   - 「〜を理解するために」「〜の能力を養うために」等の目的表現を使う

2. **課題の要件**
   - 必須要件: 最低限満たすべき条件（提出物・分量・形式）
   - 推奨要件: より高い評価を得るための条件（参考文献の活用、図表の作成等）
   - 分量の目安: レポートは2000〜4000字程度、プログラムは機能要件で指定

3. **取り組みの手順**
   - 3〜5ステップの段階的なガイダンス
   - 各ステップの目安時間は記載しない（学生によって異なるため）

4. **提出物の仕様**
   - ファイル形式（Markdown / PDF / ソースコード等）
   - 命名規則（例: `学籍番号_レポート1.md`）
   - 提出先（Moodle等）

5. **評価の観点（概要）**
   - ルーブリックの評価観点名を列挙する
   - 各観点の配点を記載する

### チームレポート固有の項目

6. **チーム構成に関する要件**
   - 役割分担の記載を求める（誰がどの部分を担当したか）
   - 個人の貢献を明示する方法（担当セクションの明記等）
   - チーム内でのレビュー・相互確認の推奨

---

## 提出物の分量目安

| 提出形式 | 分量の目安 |
|---------|----------|
| Markdown / テキストレポート | 2000〜4000字（図表除く） |
| PDF レポート | A4 3〜6ページ |
| プログラム（ソースコード） | 機能要件で指定（行数は指定しない） |
| プログラム + レポート | プログラム + 説明文1000〜2000字 |
