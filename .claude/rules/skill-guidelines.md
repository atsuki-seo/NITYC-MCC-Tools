# スキル設計ガイドライン

スキルや Agent プロンプトを作成・修正する際の設計原則。

## 具体例 vs 原則の使い分け

スキルや Agent プロンプトで具体例を書くべきかの判断基準: **期待する振る舞いが1つに固定されるか？**

### 具体例が有効（振る舞いが固定）

- **システム制約の説明**: 正解が1つしかない技術的事実（例: 特定の構文が許可プロンプトを発生させる → 回避方法は1つ）
- **API リファレンス**: 正しい呼び方が決まっている（例: 必要なフィールド名、クエリ形式）

### 具体例を避けるべき（振る舞いが不定形）

- **禁止リスト**: 「X を使うな」と具体名を列挙すると、リストにない手段で同じことをする（python3 → swift → ...）
- **許可パターンリスト**: 「以下は OK」と列挙すると、パターンマッチで類似手法を探索する動機になる

### 不定形な振る舞いを制御する方法

具体例で禁止する代わりに、以下のアプローチで正解を固定する:

1. **原則で正解を示す**: 「〜するな」ではなく「〜する」と書く（例: 「外部コマンドを使わない」→「Read → 自分で計算 → Write で出力する」）
2. **構造で選択肢を制限する**: 環境レベルで不要な手段を封じる（例: バックグラウンド Agent にして Bash 自体を使えなくする、出力フォーマットをテンプレートで固定する）

## 不要な分岐を作らない

全ケースで通用するアプローチがあるなら、条件分岐を入れずそのアプローチだけを書く。

分岐が正当なケース:
- 条件によって最適なアプローチが本質的に異なる
- 分岐しないと無駄なコストが発生する

分岐を避けるべきケース:
- 一方のアプローチが全ケースをカバーできる
- 分岐の判断自体にコストや不確実性がある
